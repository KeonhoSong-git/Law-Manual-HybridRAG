# 청킹 · 데이터 처리 상세

수집(law.go.kr) → 청킹 → 지식그래프 → 임베딩까지, 각 단계의 **실제 규칙**을 정리한다.
(코드: `data_tools/fetch_law.py`, `kg/build_v3.py`, `kg/classify_edges_v3.py`, `data_tools/embed_corpus.py`)

---

## 1. 수집 (국가법령정보센터 OPEN API)

- **검색** `lawSearch.do?target=law|admrul&query=…&type=JSON` → 문서 목록.
- **본문** `lawService.do` — 법령은 `MST=<일련번호>`, **행정규칙은 `LM=<행정규칙명>`**(이름)으로 조회(파라미터가 다름).
- HTML 직접 스크래핑은 law.go.kr이 **JS SPA**라 원문 페이지가 제목 수준만 반환(≈1.3KB) → **API(조문 구조 JSON)** 채택.
- 호출 기기 **IP/도메인 등록 + OC** 필요(미등록 시 "사용자 정보 검증 실패").

## 2. 청킹 규칙 (1 조문 = 1 청크 원칙)

### 2.1 법령 (`target=law`)
응답의 `조문단위` 리스트를 순회한다.
- **제외**: `조문여부 == "전문"`(편/장/절 헤딩), `제N조 삭제`, 빈 본문.
- **라벨/제목**: `제{조문번호}조` · `조문제목`.
- **본문**: `조문내용` + 중첩된 **항·호·목**까지 합침 — `_collect_text`가 키가 `…내용`으로 끝나는 값(`조문내용`/`항내용`/`호내용`/`목내용`)을 순서대로 수집. HTML 태그·연속 공백 정리.
  - 정의 조항(`제2조(정의)`)은 호 단위 정의("1. "신기술사업자"란 …")까지 본문에 포함 → KG 정의 추출·벡터 검색에 유리.
- **날짜**: 공포일자/시행일자 `YYYYMMDD` → `YYYY. M. D.`(build_v3 날짜 정규식이 먹는 형식).

### 2.2 행정규칙 (`target=admrul`, 법규체)
구조가 법령과 달라(`AdmRulService`, 조문단위 없음) **본문 텍스트를 조문 경계로 분할**한다.
- 본문 = `조문내용` 리스트(문자열·딕트 혼재)를 합쳐 1개 텍스트로.
- **분할 기준**: 정규식 `(?=제\d+조\s*\()` — **괄호 제목이 붙은 `제N조(제목)`** 에서만 자른다.
  - ⇒ 인라인 인용(`제7조제2항`, `제4조에서…`)에서는 **안 쪼개짐**(파편 방지).
- 파편(<15자)·`삭제` 조 제외. 분할 실패 시 전문 1청크(최대 8000자)로 보존.

> 즉 **법령은 구조 필드를 그대로 승격, 행정규칙은 평문을 규칙으로 재분할**한다.

### 2.3 매뉴얼 · 지침서 (`data_tools/fetch_manual.py`, 공문체)
law.go.kr에서 **종류가 매뉴얼/안내서/가이드/운영요령** 인 행정규칙은 본문(`조문내용`)이 비어 있고, 실제 내용은 **PDF 첨부파일**(`flDownload.do?flSeq=…`)로만 제공된다(2.1·2.2 조문 경로로는 안 잡힘).
- **검색**: `lawSearch.do`(admrul)로 제목에 매뉴얼류 키워드가 든 문서를 찾고, `lawService.do?ID=<일련번호>`의 `첨부파일링크`/`첨부파일명`에서 **`.pdf` 첨부**를 고른다(별표·서식은 후순위).
- **추출**: `pymupdf`(fitz)로 PDF 텍스트 추출. HWP에서 PDF로 변환된 정부 문서는 ToUnicode가 없는 CID 폰트가 많아 `pdfminer`로는 글자깨짐(`(cid:NNNN)`) → **MuPDF 채택**.
- **품질 게이트**: 추출 한글 글자수가 임계(기본 400) 미만이면 **스캔(이미지) PDF·깨짐**으로 보고 건너뜀.
- **청킹**: 조문 구조가 없으므로 **공문체 윈도우 청킹** — 최상위 아웃라인(`제N장`, `Ⅰ.`, `1.`) 경계로 큰 절을 나눈 뒤, 긴 절은 문장 경계로 ~900자 윈도우. 목차 점선(`@@@`)·장식 글리프·PUA 문자는 정리.
- 결과 메타: `doc_family=공문체`, `doc_type=매뉴얼`, `unit=본문`, `article_label=§N`.

> KG에서 매뉴얼은 **조문 노드 없이 문서 노드까지만**(Lean KG) — 절차 등 세부 내용은 벡터 검색이 담당한다.

### 2.4 로컬 hwpx / txt (`data_tools/ingest_hwpx.py`)
OPEN API 외에 **로컬 문서 파일**도 같은 `by_document` 형식으로 받는다(OC·인터넷 불필요).
- **hwpx** = OWPML(zip). **외부 의존 없이** `Contents/section*.xml`의 `<hp:t>`(텍스트 런)을 `<hp:p>`(단락) 단위로 추출.
- 추출한 본문을 **2.2와 동일한 `제N조(제목)` 규칙**으로 청킹. 조문 구조가 없으면(공문/매뉴얼) 전문 1청크.
- `doc_family`는 본문에 `제N조(…)`가 있으면 `법규체`, 없으면 `공문체`로 자동 판정. 제·개정일은 본문 날짜 패턴에서 추출.
- 지원: `.hwpx`, `.txt`. 구버전 `.hwp`·`.pdf`는 미지원(외부 변환 후 ingest). 표·도형·페이지순서 정밀 복원이 필요하면 별도 `hwpx_chunker`(rhwp 엔진).

## 3. 청크 형식 — `by_document/<문서>.json`

문서 1개 = 청크 dict 배열(첫 원소가 문서 메타 운반).

| 필드 | 의미 |
|------|------|
| `doc_title` | 문서명(법령명/행정규칙명/매뉴얼명) |
| `source_file` | `법령_<명>.json` / `행정규칙_<명>.json` / `매뉴얼_<명>.pdf` |
| `doc_family` | `법규체`(조문 구조 → 조문 노드 생성) / `공문체`(매뉴얼 → 문서 노드만) |
| `doc_type` | `법령` / `행정규칙` / `매뉴얼` |
| `enacted` / `last_amended` | 공포일 / 시행일 (`YYYY. M. D.`) |
| `unit` | `조`(법규체) / `본문`(공문체) (또는 분할 실패 시 `문서`) |
| `article_label` / `article_title` | `제N조` / 조문제목 (공문체는 `§N` / 빈값) |
| `text` | 조문 본문(항·호 포함) 또는 매뉴얼 본문 윈도우 |

## 4. 지식그래프 구축 — `kg/build_v3.py` (결정적, LLM 없음)

청크 → 트리플. **규칙/정규식 + `uuid5`** 로만 생성.

| 산출 | 규칙 |
|------|------|
| 문서 노드 | 1 문서 = 1 노드. 유형별 클래스(`eli:LegalResource` 등) |
| 조문 노드 | `article_label` → `eli:LegalResourceSubdivision` + `eli:has_part` |
| 정의용어 | 정의조항(`제N조(정의)`) 안의 `N. "용어"란 …` → `skos:Concept` + `skos:definition` |
| 기관 | `…위원회/심의회/협의회/이사회` 명사구(불릿·단편 제거, 최소 5자, 상위문자열 흡수) → `org:Organization` |
| 문서간 참조 | ① 「」 인용 ② 괄호 밖 **정확 제목(정규화 ≥7자)이 본문에 실재** → `dct:references` |
| 메타 | 공포/시행일 → `eli:date_publication` 등 |

**클리닝**: 깨진 제목은 파일명 최빈 세그먼트로 복원, 기관 조각(상위명의 진부분문자열) 드롭, 약칭(「전체」(이하 「약칭」)) 해소.
**IRI**: `uuid5(정규화 키)` → 재실행해도 동일.

## 5. 인용 종류분류 — `kg/classify_edges_v3.py` (LLM + 게이트)

`dct:references`(무방향) 중 **법규체→법규체**에 대해 종류를 부여.
1. **LLM 제안**: 본문을 주고 대상별 `{relation, evidence}` 받음(`based_on`/`applies`/`amends`/`cites`).
2. **게이트(코드 검증)** — 통과만 채택:
   - **키워드**: evidence에 `위임`(based_on)·`준용`(applies)·`개정`(amends) 명시될 때만 승격, 아니면 `cites`.
   - **충실성**: evidence가 원문에 실재.
   - **방향**: `_rank`(규정 0 < 요령 1 < 기준·지침 2 < 기타 3) — based_on/applies는 하위→상위만.
- 결과 대부분 `cites`로 수렴(보수적) → 잘못된 강한 관계(위임 등) 생성 위험 최소화.

## 6. 임베딩 — `data_tools/embed_corpus.py`

- 청크 `text`(≤1000자) → **BGE-M3 1024차원**(OpenAI 호환 `/embeddings`, 배치 32).
- 저장(경량): **`vectors.npy`**(float16, N×1024) + **`vectors.meta.jsonl`**(행별 `{id,doc,text}`, 벡터 행과 순서 일치).
  - JSON 풀정밀 대비 약 1/12 용량.
- `id = uuid5(NS, doc + "\x01" + text)` — 청크 결정적 식별자.
- 데모 부팅 시 정규화 numpy 행렬로 적재 → 쿼리당 코사인 1회.

## 7. 전체 흐름

```
law.go.kr OPEN API
  └ 검색→본문(법령 MST / 행정규칙 LM)
        │  fetch_law: 조문 단위 청킹(법령=구조승격 / 행정규칙=제N조 재분할)
        ▼
by_document/*.json
        │  build_v3: 규칙으로 노드·참조 트리플(결정적)        → kg.ttl
        │  classify_edges_v3: 인용 종류분류(LLM 제안+게이트)  → 오버레이
        │  embed_corpus: BGE-M3                              → vectors.npy + meta
        ▼
demo/data/{kg.ttl, vectors.npy, vectors.meta.jsonl}  →  데모(:8800)
```
